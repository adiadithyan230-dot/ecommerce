Dataset Description

Expected Schema

The application relies on a specific set of columns to perform full analytical functions. The upload validator automatically maps variations of column headers to these standard names:

Column Name Data Type Description Critical for Analysis
order_id Numeric / String Unique identifier for the transaction. Yes
customer_name String Name or ID of the purchasing customer. Used for returning customer rates. Yes
product String Name of the item purchased. Used for Product Insights. Yes
category String Broader classification of the product (e.g., Electronics, Clothing). Yes
region String Geographic location of the sale (e.g., North America, EU). Yes
quantity Numeric Number of units purchased in the transaction. Yes
revenue Numeric (Float) Total monetary value of the transaction. Yes
payment_method String Method used to pay (e.g., Credit Card, PayPal). Optional
order_status String Current state of order (Delivered, Cancelled, Returned). Yes (For Risk Alerts)
order_date Date (YYYY-MM-DD) Timestamp of the transaction. Vital for forecasting and trends. Yes

Validation & Auto-Correction

Our Pandas-based validation engine handles dirty data in the following ways:

  • Header Stripping: Leading/trailing spaces and varying capitalization (e.g., " Total Sales ") are normalized to `revenue`.
  • Type Casting: If `revenue` contains strings (like "$500"), the upload validator flags it. If castable, Pandas forces it to numeric.
  • Date Parsing: Invalid date strings like "32-13-2023" are identified and reported via `pd.to_datetime(errors='coerce')`.
  • Negative Value Checks: Transactions with negative revenue or quantity trigger validation warnings, prompting the user to correct their source file.
